# Java Rate Limiter Verification Report

## Scope

This Java project was deleted and recreated from scratch to remove stale
generated output and shallow pass signals.

The goal is not to pretend every XOLoop mode can run without an agent. The goal
is to prove every locally-verifiable Java integration path with real commands,
and to label anything agentic as requiring an external audit.

## Architecture

```text
caller key + timestamp
        |
        v
bounded bucket map
        |
        v
refill by elapsed time
        |
        v
token available? ---- yes ----> allowed decision
        |
        no
        v
rejected decision with retry-after time
```

We need a request identity (key) connected to a small memory store (bounded
HashMap of token buckets) so each caller gets an independent limit without
letting arbitrary keys grow memory forever.

## Agentic Modes

Build proposal generation, Polish, Autoresearch, and full Audit fixing are
agentic. They require live subagents or API-backed proposers to be honest. This
fixture now includes a provider-level Java check that proves Build, Polish,
Autoresearch, Audit, and Overnight invoke the shared live-agentic provider
socket and record evidence. The test provider is deterministic, so it proves the
plugin wiring and contracts; production proof still requires a real subagent/API
provider.

## External Audit Follow-Up

The external audit found one blocking plugin bug:

- Benchmark execution could pass when a command printed matching JSON and then
  exited nonzero.

That is fixed in `benchmark_runner.cjs`; a regression test now proves a nonzero
exit is a benchmark violation even if stdout matches.

The audit also noted residual limits:

- Some repo-wide all-mode tests are prompt/summary/harness checks, not live
  agentic end-to-end runs.
- Java static audit is a heuristic, not a full parser-backed Java analyzer.
- The no-mocks check proves the Java shell dogfood uses real commands, but it is
  not a proof that every plugin mode can run agentically without credentials.
